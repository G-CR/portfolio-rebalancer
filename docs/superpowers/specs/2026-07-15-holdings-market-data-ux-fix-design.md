# 持仓行情与表格交互修复设计

## 1. 背景

持仓与成本维护页目前存在三个直接影响使用的问题：

1. 国内 ETF 行情自动获取失败，持仓页只能显示“行情数据不完整”。
2. 数字列标题左对齐，而对应数值右对齐，视觉上像是列错位。
3. 行操作菜单位于横向滚动容器内部，展开后被容器裁切，需要滚动才能看到完整菜单。

本次修复保持现有“组合校准台”视觉体系和高密度台账结构，不改变持仓、成本和盈亏计算口径。

## 2. 根因

### 2.1 行情获取

后端实现了 `AkshareProvider`，默认国内行情优先级也是 AKShare，但运行依赖没有包含 `akshare`。因此每次调用都会抛出 `ModuleNotFoundError`，随后继续尝试未配置的 Tushare。当前数据源选择错误只保留最后一次失败，最终页面看到的是“Tushare 未配置”，而不是 AKShare 缺失或请求失败的真实原因。

持仓创建成功后，前端只刷新持仓和分析查询，没有立即触发行情刷新。定时 worker 尚未运行到下一个刷新时间前，新持仓必然处于缺失状态。

### 2.2 表格对齐

数据单元格通过 `.numeric` 右对齐，但表头统一左对齐。列宽本身由固定表格布局控制，用户看到的错位主要来自同一数字列采用了不同的文本对齐基准。

### 2.3 菜单裁切

菜单使用 `position: absolute`，定位祖先位于 `overflow-x: auto` 的表格容器中。浏览器会将另一轴的溢出也纳入滚动裁切区域，菜单无法越过容器边界。

## 3. 目标

- 国内 ETF 默认行情源可在正式 Docker 镜像中运行。
- 新增持仓或进入存在缺失行情的持仓页时，自动尝试一次刷新。
- 刷新中使用中性反馈；失败后展示可理解的原因以及重试、手动录入入口。
- 表头与数据采用一致的列宽和对齐规则。
- 操作菜单不受表格容器裁切，并能根据可用空间向上或向下展开。
- 保持键盘操作、窄屏布局和现有抽屉工作流。

## 4. 非目标

- 不增加盘中实时推送或高频轮询。
- 不改变行情刷新计划和快照计划。
- 不改变数据源优先级编辑方式。
- 不重做整个持仓页面或移动端信息架构。
- 不实现券商交易或已实现盈亏。

## 5. 行情刷新设计

### 5.1 运行依赖

后端正式与开发镜像均从同一个锁文件安装依赖。将兼容 Python 3.13 的 `akshare` 固定到项目依赖并更新 `uv.lock`，保证 `AkshareProvider.fetch_price()` 在容器中可导入并执行。

### 5.2 数据源失败诊断

`ProviderRegistry` 记录每个尝试过的数据源及安全失败类别。选择最终诊断时遵循：

1. 已配置数据源的请求失败或响应无效优先于备用数据源未配置。
2. 诊断包含数据源名称和安全类别，不包含 URL、密钥、原始响应或异常正文。
3. `MarketData` 仍只写入一条本次刷新失败记录，但 `error_summary` 能说明主要失败来源和备用来源状态。

示例：`akshare: provider_request_failed; tushare: provider_not_configured`。

### 5.3 持仓页自动刷新

持仓页在分析接口返回 `PORTFOLIO_DATA_INCOMPLETE` 且缺少价格或汇率时，每次页面挂载周期最多自动刷新一次。刷新调用沿用 `POST /api/market-data/refresh`，不新增接口。

状态流转：

1. **缺失且尚未尝试**：自动开始刷新。
2. **刷新中**：显示“正在获取行情”，保留持仓与成本字段，不显示红色错误。
3. **刷新成功且数据完整**：重新获取分析数据，提示自动消失并展示市值与盈亏。
4. **刷新完成但仍缺失**：显示失败来源和安全原因，提供“立即重试”和“手动录入”。
5. **刷新请求异常**：显示通用网络或服务错误，并保留相同操作入口。

“立即重试”允许用户显式再次调用刷新，不受自动刷新一次的限制。“手动录入”导航到数据源页并定位对应 `price:<symbol>` 或 `fx:<pair>` 项。

### 5.4 查询失效

刷新成功后统一失效以下查询：

- `market-data`
- `portfolio-analytics`

持仓创建成功后仍立即关闭抽屉；持仓页观察到新的缺失项后自动刷新，避免让网络行情请求阻塞持仓保存。

## 6. 表格对齐设计

- 使用 `colgroup` 作为桌面表格的唯一列宽定义，表头和数据行共享同一组列。
- 标的、账户、币种列左对齐。
- 份额、成本价、成本汇率、当前价、当前汇率、市值和浮动盈亏统一右对齐。
- 操作列保持右对齐且宽度稳定，按钮状态不能改变列宽。
- 现有 1100px 和 759px 响应式隐藏规则保留；隐藏列同时作用于对应 `col`、`th` 和 `td`。

## 7. 智能浮层菜单

采用已确认的方案 A。

- 菜单通过 React Portal 渲染到 `document.body`，使用 `position: fixed`。
- 打开时读取触发按钮的 `getBoundingClientRect()`。
- 默认向下展开；下方空间不足且上方空间更充足时向上展开。
- 菜单右边缘与触发按钮右边缘对齐，并限制在视口安全边距内。
- 窗口缩放、页面滚动和表格横向滚动时重新计算位置。
- 点击菜单外部、按 `Escape`、执行命令或触发按钮再次点击时关闭。
- 打开后聚焦第一个菜单项；关闭后将焦点还给触发按钮。
- 菜单视觉继续使用现有边框、阴影、字号和颜色，不引入新的设计语言。

## 8. 错误文案

刷新中：

> 正在获取 563020 行情…

失败：

> 563020 行情获取失败。AKShare 请求失败，备用 Tushare 未配置。

操作：

- `立即重试`
- `手动录入`

不直接向用户显示 Python 异常类、内部 URL 或第三方原始响应。

## 9. 测试与验收

### 9.1 后端

- 容器中可以导入 `akshare`。
- AKShare 返回 ETF 行情时可生成合法报价。
- 主数据源请求失败、备用源未配置时，诊断保留主失败原因。
- 所有数据源均失败时不覆盖最近有效行情。
- 安全诊断不包含凭据或请求 URL。

### 9.2 前端单元测试

- 缺失行情时自动刷新一次，React StrictMode 下不重复请求。
- 刷新中显示中性状态。
- 刷新成功后重新请求分析并移除提示。
- 失败后显示“立即重试”和“手动录入”。
- 数字表头使用右对齐类和共享列定义。
- Portal 菜单可以向上、向下定位，支持外部点击和 `Escape`。

### 9.3 浏览器验收

- 桌面宽度下表头与每列数值对齐。
- 第一行和最后一行菜单均完整可见，无需纵向滚动表格。
- 横向滚动表格后菜单仍贴合触发按钮。
- 390px 宽度下现有详情展开和操作入口可用。
- 键盘可以打开菜单、执行命令并恢复焦点。

## 10. 完成标准

- `563020` 在 Docker 环境中可以通过默认国内数据源刷新到有效价格，或在真实网络失败时显示准确、安全的原因。
- 新持仓无需等待定时任务即可开始自动获取行情。
- 数字标题和数值在支持的桌面宽度下共用右对齐基准。
- 菜单不会被表格、页面底部或视口边缘遮挡。
- 后端、前端和 Playwright 全套测试通过。
